# Supply chain: SBOM and vulnerability scanning

## What is produced

On every push, the `supply-chain` CI job produces and uploads:

- `sbom-llmd-packages.spdx.json` and `sbom-llmd-packages.cdx.json`:
  the Software Bill of Materials of the packages this project builds
  and ships (the `llmd-*` debs and the repackaged llmfit binary),
  generated by syft in SPDX and CycloneDX formats.
- `installed-packages.tsv`: the full package inventory of a system
  actually installed from the ISO by the install test, captured over
  SSH from the running machine. This is ground truth, not a build-time
  estimate, and it covers the release ISOs too: they install the same
  package set, only the installer screens differ.
- `cve-report.json`: the trivy vulnerability report computed against
  that installed inventory.

On release tags the two SBOM files and the inventory are attached to
the GitHub release.

Tool versions are pinned in the workflow (syft v1.46.0, trivy
v0.72.0) and bumped deliberately, like every other pin in this
project.

## The gate

The scan fails the build when it finds a CRITICAL vulnerability with
a fix available. Everything else is reported but not fatal:

- Unfixed vulnerabilities are inherited from the Ubuntu base and
  cannot be acted upon here; they are visible in the report and in
  the Ubuntu CVE tracker.
- HIGH and below with a fix are expected to be picked up by
  unattended-upgrades on installed systems and by the next base image
  refresh in CI; they do not block development.

## Triage policy

When the gate fires:

1. Prefer the real fix: refresh or pin the affected package, or bump
   the base image, in the same PR.
2. If the finding does not apply to SibillaOS (component not used,
   code path unreachable, mitigation already in place), record an
   exception in `.trivyignore` with a comment stating the reason and
   a review-by date. Exceptions without both do not pass review.
3. If the fix needs time, open an issue titled `CVE-...`, link it in
   the `.trivyignore` comment, and treat the review-by date as a
   deadline, not a suggestion.

Exceptions are re-reviewed at every release: release notes must not
ship with an expired review-by date in `.trivyignore`.
